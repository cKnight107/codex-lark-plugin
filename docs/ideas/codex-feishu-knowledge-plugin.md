# Codex 飞书知识检索插件

## Problem Statement

How might we 为一个 5-20 人、以 Codex 为主要协作入口的团队，提供一个能稳定读取和检索飞书文档的插件，让团队成员在 Codex 中就能理解项目上下文、定位相关文档，并逐步支撑项目与任务协作？

## Recommended Direction

第一版做一个“文档驱动但结构化索引支持”的 Codex 插件。

飞书文档仍然是团队编辑和协作界面，但插件后端维护一份最小结构化索引，包括文档 ID、标题、项目归属、类型、更新时间、作者、摘要和链接。

插件第一版不做复杂任务流，不做自动分发，不做自动写代码。它只把一件事做到可靠：让用户和 Codex 可以按项目、关键词、更新时间和文档类型，正确找到相关文档，并快速得到摘要。

这条路线的好处是启动快、价值清晰、风险可控，而且天然能为后续“Codex 自动注入上下文”做基础设施铺垫。

## Key Assumptions to Validate

- [x] 团队的飞书文档可以被相对稳定地映射到项目维度
  验证方式：抽 2-3 个真实项目，看是否能给现有文档打上统一 `project_id`
- [x] 团队真正高频需求是“找文档”和“理解文档”，而不是先要任务流
  验证方式：访谈 3-5 个成员，记录他们最近一周的真实协作阻塞点
- [x] 关键词 + 元数据过滤 + 摘要，已经足够支撑第一版正确率
  验证方式：用一批真实查询做 Top 3 命中测试
- [x] 团队愿意接受“文档继续在飞书维护，查询主要在 Codex 完成”
  验证方式：让 1 个小团队试用一周，看是否形成使用习惯

## MVP Scope

第一版只做 5 个能力：

- `list_project_docs(project)`
- `search_docs(query, project?)`
- `get_doc_summary(doc_id | url)`
- `list_recent_docs(project?, days=7)`
- `compare_doc_changes(doc_id, from?, to?)`

后端只维护最小索引字段：

- `doc_id`
- `title`
- `project_id`
- `doc_type`
- `author`
- `updated_at`
- `summary`
- `url`

## Not Doing

- 不做任务自动分发，因为这会把问题从“检索”扩展成“流程编排”
- 不做完整任务系统，因为第一版目标不是替代 Jira 或飞书任务
- 不做复杂问答 Agent，因为文档命中率和索引质量还没验证
- 不做自动改代码，因为这不是当前最核心用户价值
- 不做大而全协作平台，因为团队现在要的是可用入口，不是新系统

## Open Questions

- 项目归属由谁维护，自动推断还是人工标注？
- 飞书文档分类是否需要最小规范，例如 `需求`、`方案`、`会议纪要`、`任务`、`复盘`？
- Codex 插件调用后端的方式是什么，HTTP 服务、MCP，还是插件内嵌能力？
- 第一版是否需要“最近更新订阅”，还是先停在被动查询？
